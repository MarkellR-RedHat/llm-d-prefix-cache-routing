Chapter One

The Problem

Every time an LLM processes a prompt, it computes a massive matrix of attention states -- the KV cache. When two requests share the same beginning, that computation is identical -- yet standard load balancers force every pod to redo it from scratch.

Round-Robin Routing

Request 1 goes to Pod A, which computes the full KV cache. Request 2 -- with the exact same system prompt -- goes to Pod B, which has to compute it all over again. The cache sitting in Pod A is wasted. Multiply this across every request, and you are burning GPU cycles recomputing attention states that already exist somewhere in your fleet.

Cache-Aware Routing

The EPP (Endpoint Picker Policy) knows what each pod has cached. When Request 2 arrives with the same prefix, the EPP routes it to Pod A -- the pod that already has the cache. Pod A skips the entire prefill phase and goes straight to generating tokens. Zero redundant compute. The longer the shared prefix, the bigger the savings.

Chapter Two

The Insight

The KV cache is not a temporary artifact -- it is a reusable asset. When routing is aware of what each pod has cached, the same computation never has to happen twice.

How Long Does the Cache Last?

The most common question people ask. The answer: there is no timer. The prefix cache lives in GPU memory and persists until something pushes it out.

Not time-based -- memory-based

vLLM uses LRU (least recently used) eviction. When GPU memory fills up with cached blocks from many different prompts, the blocks that have not been accessed in the longest time get evicted first. Under low traffic with consistent system prompts, cached blocks can persist indefinitely -- hours, days, as long as the pod is running.

What clears the cache

Three things evict cached blocks: memory pressure from diverse prompts (LRU eviction), pod restarts (cache starts empty), and GPU memory limits. There is no TTL, no expiration timer, no scheduled flush. If your workload keeps hitting the same system prompt, those blocks stay warm.

Why this matters for routing: Cache-aware routing compounds the benefit. By concentrating requests with the same prefix onto the same pod, llm-d keeps that pod's cache hot for exactly the tokens that matter. The more consistent your prompt patterns, the higher your cache hit rate -- and the longer those blocks persist in GPU memory without eviction.

Where It Matters Most

Prefix-cache-aware routing delivers the biggest gains in workloads with high prefix overlap. These three patterns are where the math works hardest in your favor.

Multi-Turn Conversations

Each turn in a conversation builds on the previous one. Turn 3's prompt contains all of turns 1 and 2 as prefix. Routing to the same pod means turns 1-2 are already cached -- only the new user message needs prefilling.

Turn N reuses 100% of turns 1 through N-1

Agentic Workflows

Agent frameworks send parallel requests that all share the same system prompt. The first call computes the cache; every subsequent parallel call to the same pod skips the system prompt prefill entirely. 10 parallel tool calls = 1 cold prefill + 9 cache hits.

N parallel calls = 1 prefill + (N-1) cache hits

Long-Context Documents

When users ask multiple questions about the same document, the document tokens are identical across requests. A 32K-token document cached on Pod A means every follow-up question skips 32K tokens of prefill computation -- that is where the 47.5x speedup comes from.

32K cached tokens = 3,700ms to 79ms TTFT

Chapter Four

The Impact

Four steps from request to routed response. The EPP makes this decision in microseconds, before the request ever reaches a vLLM pod.

1

Request Arrives

A chat completion request hits the Gateway API. The EPP intercepts it and extracts the token prefix -- the sequence of tokens that make up the prompt (system message + conversation history).

2

Query Cache State

The EPP maintains a near-real-time map of what each pod has cached. It checks every replica's cached prefix blocks against the incoming request's token sequence.

3

Score by Overlap

Each pod gets a score based on the longest prefix match -- how many contiguous token blocks from the start of the request are already in that pod's KV cache. More blocks matched = higher score.

4

Route to Best Match

The request is routed to the highest-scoring pod. On a full cache hit, the pod skips the entire prefill phase -- it already has the attention states computed and goes straight to generating new tokens.

Common Questions

The questions people ask most about prefix-cache-aware routing -- from conference halls to Slack channels.

Can the cache be shared across pods?

No. The KV cache lives in each pod's GPU memory -- Pod A cannot read blocks from Pod B's GPU. This is exactly why routing matters: if you cannot share the cache, send the request to the pod that already has it.

What happens when a pod restarts?

Its cache is gone. New pods start cold. The EPP discovers them automatically and routes based on load until they build up cache. Existing pods keep their warm caches while new ones ramp up.

Does this add latency?

The EPP scoring adds microseconds -- it is a hash comparison across a small set of pods. The routing cost is invisible. The savings from skipping prefill (hundreds of ms to seconds) are not.

Does it work with any model?

Any model served by vLLM with automatic prefix caching enabled (the default). Llama, Mistral, Qwen, Falcon -- any transformer. The cache operates at the KV block level, a property of attention, not specific model weights.

What is the minimum prefix length?

vLLM caches in blocks of 16 tokens. You need at least 16 shared tokens to see a hit. In practice, even a short system prompt like "You are a helpful assistant" exceeds this once tokenized.

What about different system prompts?

They cache independently. User A's requests cluster on one pod, User B's may cluster on another. Multiple distinct caches coexist on a single pod as long as GPU memory allows.

What if all pods are cold?

The EPP falls back to load-based routing -- fewest active requests wins. That first request computes the cache, and all subsequent requests with the same prefix benefit from it.

Does it work with LoRA adapters?

Yes. Prefix caching operates on the base model's KV blocks. LoRA modifies weights but the caching mechanism is the same. Requests with the same base prefix still get cache hits regardless of which adapter is applied.

How much GPU memory does the cache use?

vLLM uses whatever GPU memory is not active for inference -- no separate allocation or config. On an H200 with 141GB serving a 70B model in FP8 (~70GB), there is substantial room for cached blocks. vLLM manages this automatically.

Do I need to change my application code?

No. Cache-aware routing is a routing-layer optimization. Your prompts, model, and application stay the same. Swap in llm-d as your inference gateway and cache reuse happens automatically -- no SDK changes, no prompt modifications.

Prefix-cache-aware routing turns a simple observation -- that many requests share the same beginning -- into a powerful optimization. By routing with awareness of what each pod already knows, llm-d transforms redundant GPU compute into instant cache hits. The result is faster time-to-first-token, lower cost per request, and infrastructure that gets smarter the more consistent your workloads become.
